# Phase 4 -- UI Design Contract

> Visual and interaction contract for Multi-Medium Expansion. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none -- Obsidian plugin, not a web app |
| Preset | not applicable |
| Component library | Obsidian Plugin API (`Setting`, `Notice`, `Menu`) |
| Icon library | none -- text-only status indicators per Phase 1 D-01 |
| Font | Obsidian theme font (inherited via `var(--font-text)`) |

**Rationale:** Carries forward from Phase 3. This is an Obsidian plugin. All UI uses Obsidian's Setting API, vanilla DOM manipulation, and CSS with Obsidian's CSS custom properties. No external component library. Styles live in `styles.css` at the project root.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline gaps between status text segments |
| sm | 8px | Setting item internal padding, label margins |
| md | 16px | Default element spacing (Obsidian Setting API) |
| lg | 24px | Space between settings heading groups |
| xl | 32px | not used in this phase |
| 2xl | 48px | not used in this phase |
| 3xl | 64px | not used in this phase |

Exceptions:
- Canvas node placement gap is 40px (Phase 2 `DEFAULT_SPATIAL_CONFIG.placementGap`). Spatial positioning, not UI spacing.
- Image placeholder node dimensions are 512x512px (square, matching the 1:1 aspect ratio of generated images). This is content sizing, not spacing.

**Source:** Phase 3 UI-SPEC spacing scale carried forward unchanged.

## Color

This plugin uses Obsidian's CSS custom properties exclusively. The 60/30/10 split maps to Obsidian's existing theme structure:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `var(--background-primary)` | Canvas background, settings background (inherited) |
| Secondary (30%) | `var(--background-secondary)` | Settings tab sidebar, status bar background (inherited) |
| Accent (10%) | `var(--interactive-accent)` | Pulsing border on streaming/placeholder nodes |
| Destructive | `var(--text-error)` | Error states: image generation failure text, API errors |
| Warning | `var(--text-warning)` | Runware API key missing notice |
| Muted | `var(--text-muted)` | Status bar idle state, placeholder "Generating image..." text |

Accent reserved for:
- Pulsing border animation on pre-allocated text/code/mermaid nodes awaiting first token (carried from Phase 3)
- Pulsing border animation on image placeholder nodes awaiting Runware response

### AI Node Color (carried from Phase 3)

| Property | Default Value | Configurable |
|----------|---------------|-------------|
| AI node color | `"6"` (canvas preset color 6) | Yes, via settings |
| AI node border during streaming/placeholder | `var(--interactive-accent)` with pulsing animation | No |

All four medium types (text, code, mermaid, image) use the same AI node color preset. Visual distinction between mediums comes from the content itself (markdown, fenced code, mermaid diagram, embedded image file), not from different node colors.

### Empty/Edge States

| State | What the User Sees |
|-------|-------------------|
| No image generated (Claude chose not to) | No image node appears. Only text/code/mermaid nodes appear as selected by Claude. No user-facing message. |
| Image placeholder waiting for Runware | Square node with pulsing accent border. Node text reads `Generating image...` in muted color. |
| Image generation succeeds | Placeholder swapped for file node displaying the rendered image. Pulsing animation removed. |
| Image generation fails (network/auth) | Placeholder text replaced with `Image generation failed`. Pulsing animation removed. Notice shown. |
| Mermaid buffering (waiting for complete diagram) | Empty node with pulsing accent border. No intermediate text. |
| Mermaid flush (diagram complete) | Complete `\`\`\`mermaid` code block flushed into node in one step. Obsidian renders the diagram natively. Pulsing animation removed. |
| Mermaid incomplete (stream error mid-diagram) | Partial mermaid content flushed with `%% (incomplete -- generation was interrupted)` comment appended. |
| Code streaming | Code appears line by line inside a fenced code block with language tag. Obsidian applies syntax highlighting progressively. |
| Runware API key missing | Image placeholder shows `Image generation failed`. Notice: `Canvas AI: Image generation failed. Check your Runware API key in Settings.` |

### Destructive Actions

| Action | Confirmation Approach |
|--------|----------------------|
| Reset taste profile | Obsidian `Notice` with 10-second display (carried from Phase 3) |
| Budget override | No confirmation -- single toggle in settings, reversible (carried from Phase 3) |

No new destructive actions in Phase 4. Image generation is additive (creates files in vault). Generated images can be deleted by the user through normal vault file management.

## Component Inventory

These are the UI elements Phase 4 adds or modifies. Obsidian Plugin API components only.

### Status Bar States (unchanged from Phase 3)

```
StatusBarState type: 'idle' | 'thinking' | 'streaming' | 'budget' | 'error'
```

No new status bar states. The existing `streaming` state covers all medium types (text, code, mermaid, image). Image generation happens asynchronously after the Claude stream completes that node, so the status bar reflects the Claude stream state, not the Runware request state.

### Settings Tab Extension

One new field added to the existing settings tab, placed after the "Taste Profile" section:

**Image Generation** (heading)
- Image save location: text input, default `canvas-ai-images`, validated to be a valid vault path

### Canvas Node Visual States by Medium Type

| Medium | Pre-allocation Size | Placeholder Visual | Content Delivery | Final Visual |
|--------|--------------------|--------------------|-----------------|-------------|
| text | 300w x 200h | Pulsing accent border, empty | Progressive streaming (200-300ms buffer) | Rendered markdown in text node |
| code | 400w x 250h | Pulsing accent border, empty | Progressive streaming (200-300ms buffer) | Fenced code block with syntax highlighting |
| mermaid | 400w x 300h | Pulsing accent border, empty | Buffered -- nothing shown until complete | Complete mermaid diagram rendered by Obsidian |
| image | 512w x 512h | Pulsing accent border + "Generating image..." text | N/A (async Runware request) | File node with embedded image |

**Source for sizes:** MMED-09 specifies "code wider, images square." Text uses Phase 3 default (300x200). Code is wider (400x250) for readability. Mermaid is wider and taller (400x300) for diagram space. Image is square (512x512) matching Runware's 1024x1024 output scaled to a reasonable canvas display size.

All new CSS rules to add to `styles.css`:

```css
  color: var(--text-muted);
  font-style: italic;

  align-items: center;
  justify-content: center;
  height: 100%;

```

**Note:** The `canvas-ai-node--streaming` class and `canvas-ai-pulse` keyframes from Phase 3 are reused without modification for all medium type placeholders. The new `canvas-ai-node--image-placeholder` class is additive -- applied alongside `canvas-ai-node--streaming` on image nodes to center the "Generating image..." text.

### Pulsing Animation Specification (carried from Phase 3)

- Duration: 1.5 seconds per cycle
- Easing: ease-in-out
- Effect: box-shadow pulsing from 2px to 4px spread, accent color fading to 40% opacity at peak
- Applied to: all pre-allocated nodes (text, code, mermaid, image) via `canvas-ai-node--streaming` class
- Removed when: first token arrives (text/code), content flushed (mermaid), or image resolves/fails (image)
- Must not cause layout shift (box-shadow only, no border-width changes)

## Interaction Contracts

### Generation Flow State Machine (extended from Phase 3)

```
IDLE --[canvas event]--> DEBOUNCING --[timeout]--> THINKING --[first token]--> STREAMING --[complete]--> IDLE
                                                       |                            |
                                                       v                            v
                                                   ERROR --[retry]--> THINKING   ERROR --[retry]--> STREAMING
                                                       |
                                                       v
                                                   IDLE (after max retries)

BUDGET_EXCEEDED interrupts at THINKING stage:
THINKING --[budget check fail]--> BUDGET --[override]--> THINKING

IMAGE GENERATION (parallel, non-blocking):
STREAMING --[</node type="image"> detected]--> fire async Runware request
  Runware request --[success]--> swap placeholder for file node
  Runware request --[failure]--> update placeholder with error text + Notice
  (Claude stream continues regardless of Runware result)
```

### Multi-Medium Node Sequence (D-01 through D-12)

1. **Claude streams response** with typed `<node type="...">` tags. Up to 4 nodes total, each a distinct type.

2. **For text nodes (`<node type="text">`):**
   - Pre-allocate 300x200 text node at computed placement with AI color and `canvas-ai-node--streaming` class
   - On first token: remove `canvas-ai-node--streaming` class, stream content progressively
   - On `</node>`: finalize node content

3. **For code nodes (`<node type="code" lang="...">`):**
   - Pre-allocate 400x250 text node at computed placement with AI color and `canvas-ai-node--streaming` class
   - On first token: remove `canvas-ai-node--streaming` class
   - Wrap streaming content in fenced code block: `` ```{lang}\n{content}\n``` ``
   - Stream progressively -- user sees code written line by line with syntax highlighting
   - On `</node>`: finalize node content with complete fenced block

4. **For mermaid nodes (`<node type="mermaid">`):**
   - Pre-allocate 400x300 text node at computed placement with AI color and `canvas-ai-node--streaming` class
   - Buffer all content -- do NOT flush to node during streaming. Node stays empty with pulsing border.
   - On `</node>`: flush complete content as `` ```mermaid\n{content}\n``` `` in one step. Remove `canvas-ai-node--streaming` class. Obsidian renders the diagram.
   - On stream error: flush partial content with `%% (incomplete -- generation was interrupted)` appended. Remove `canvas-ai-node--streaming` class.

5. **For image nodes (`<node type="image">`):**
   - Pre-allocate 512x512 text node at computed placement with AI color, `canvas-ai-node--streaming` class, and `canvas-ai-node--image-placeholder` class
   - Set node text to `Generating image...`
   - On `</node>`: extract the image prompt text. Fire Runware request asynchronously.
   - Claude stream continues immediately to next node (non-blocking per D-08).
   - On Runware success: remove placeholder text node, create file node at same position with the saved image. Add AI node color. Track new file node ID in `aiNodeIds`.
   - On Runware failure: replace placeholder text with `Image generation failed`. Remove pulsing classes. Show Notice.

### One-Per-Type Enforcement (D-02)

A `Set<string>` tracks which medium types have appeared in the current generation. Before creating each new node:
- If the type already exists in the set: skip this node (do not create)
- If 4 types have been created: skip all remaining nodes
- Otherwise: add type to set, create the node

This replaces Phase 3's 1-3 same-type node cap with a max-4-distinct-type model.

### Placeholder-to-File-Node Swap Sequence (image only)

1. Runware returns base64 image data
2. Decode base64 to `Uint8Array` via `atob()`
3. Save to vault: `vault.createBinary('canvas-ai-images/{date}_{uuid-short}.png', buffer)`
4. Within `suppressEvents()` block:
   a. Remove placeholder text node from canvas
   b. Create file node at same position via `canvas.createFileNode()`
   c. Set AI node color on file node
   d. Track file node ID in `aiNodeIds`
5. Call `canvas.requestSave()`

Both remove and create must happen within a single `suppressEvents()` call to prevent visual glitch (brief flash of empty space).

### Error Recovery (extended from Phase 3)

| Error Type | User Feedback | Recovery |
|------------|---------------|----------|
| Runware auth error | Notice: `Canvas AI: Runware API key is invalid. Update it in Settings > Canvas AI.` Placeholder text: `Image generation failed`. | No retry. User must fix Runware API key. |
| Runware network error | Notice: `Canvas AI: Could not connect to Runware. Image generation unavailable.` Placeholder text: `Image generation failed`. | No retry for this generation. Next generation will attempt reconnection. |
| Runware timeout (>30s) | Placeholder text: `Image generation failed`. No Notice (silent failure for timeouts). | No retry. |
| Mermaid stream interrupted | Partial mermaid flushed with `%% (incomplete)` comment. No Notice. | Node retains partial content. User can edit or delete. |
| Code stream interrupted | Partial code preserved in node with whatever was received. No Notice. | Node retains partial content. |
| Claude API errors | Same as Phase 3 (auth, rate limit, server, network -- see Phase 3 UI-SPEC) | Same retry logic as Phase 3 |
