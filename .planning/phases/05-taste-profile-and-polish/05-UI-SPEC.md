# Phase 5 -- UI Design Contract

> Visual and interaction contract for Phase 5: Taste Profile and Polish.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Obsidian plugin -- vanilla DOM) |
| Preset | not applicable |
| Component library | Obsidian Setting API (`new Setting()`) for settings tab; raw DOM for canvas nodes |
| Icon library | Obsidian built-in icons via `setIcon()` (Lucide subset) |
| Font | Obsidian theme-controlled (`--font-text`, `--font-monospace`) |

**Context:** This is an Obsidian plugin. All UI surfaces inherit from the active Obsidian theme. The plugin does NOT ship its own design system, color palette, or typography scale. Visual contracts are expressed as Obsidian CSS custom properties and pixel values for canvas-specific layout.

---

## Spacing Scale

Canvas placement spacing (used by `computeEdgeAlignedPlacements`):

| Token | Value | Usage |
|-------|-------|-------|
| placement-gap | 40px | Gap between trigger node's right edge and first generated node (D-09). Gap between stacked generated nodes (D-10). Collision detection buffer around all nodes. |
| companion-gap | 24px | Gap between code node and its companion render node (D-12). Tighter than placement-gap because the two are visually paired. |
| vertical-stack-gap | 40px | Gap between vertically stacked generated nodes in edge-aligned placement (D-10). Matches placement-gap for visual consistency. |

**Source:** Existing `DEFAULT_SPATIAL_CONFIG.placementGap` is 40px (from `src/spatial/types.ts`). Companion gap is at Claude's discretion per CONTEXT.md -- set to 24px for visual pairing without crowding.

Exceptions:
- Companion nodes use 24px gap instead of 40px to visually associate with their parent code node.
- Settings tab spacing is entirely controlled by Obsidian's `Setting` class -- no custom spacing needed.

## Typography

All typography is theme-controlled. The plugin does NOT declare its own font sizes.

| Role | Value | Source |
|------|-------|--------|
| Settings labels | `var(--font-ui-medium)` at theme-controlled size | Obsidian Setting API |
| Settings descriptions | `var(--font-ui-small)` at theme-controlled size | Obsidian Setting API |
| Canvas node body text | `var(--font-text)` at theme-controlled size | Obsidian canvas renderer |
| Code node monospace | `var(--font-monospace)` at theme-controlled size | Obsidian fenced code renderer |
| Status bar text | `var(--font-ui-smaller)` at theme-controlled size | Obsidian status bar |

**Companion node iframe content:** HTML/CSS/JS previews render inside a sandboxed iframe. The iframe has its own document -- it does NOT inherit Obsidian's typography. No font contract is needed for iframe content because it is user-generated code.

## Companion Node Visual Contract

### Sizing

| Node Type | Width | Height | Rationale |
|-----------|-------|--------|-----------|
| HTML companion | Same as parent code node width | Same as parent code node height | Mirror sizing for visual pairing |
| Mermaid companion | Same as parent code node width | Same as parent code node height | Mirror sizing for visual pairing |
| SVG companion | Same as parent code node width | Same as parent code node height | Mirror sizing for visual pairing |

**Positioning (D-12):** Companion node is placed to the RIGHT of its parent code node, separated by `companion-gap` (24px). Top edges aligned. The companion occupies the position immediately right of the code node, NOT right of the trigger node.

**Visual relationship:** Companion nodes share the same canvas color preset as the parent code node (color "6" / Purple by default). The HTML companion gets a `.canvas-ai-companion--html` class for iframe-specific styling (white background, border).

### Iframe Styling (HTML/CSS/JS companion)

```css
/* Phase 5: Companion render node for HTML/CSS/JS */
.canvas-ai-companion--html .markdown-rendered {
  padding: 0;
  overflow: hidden;
}

.canvas-ai-companion--html iframe {
  width: 100%;
  height: 100%;
  border: none;
  background: var(--background-primary);
}
```

### Companion Identification

Companion nodes are identified via `unknownData.companionOf` property set to the parent code node's ID. This enables:
- Re-injection of iframes on canvas reload
- Visual linking between code and companion nodes
- Preventing companion nodes from triggering generation (added to `aiNodeIds` set)

## Edge-Aligned Placement Visual Contract

### Layout Rules (D-09, D-10, D-11)

```
Trigger Node          Generated Node 1
+----------------+    +----------------+
|                | -- |   (text)       |
|   [user node]  | 40 |                |
|                | px |                |
+----------------+    +----------------+
                      +--- 40px gap ---+
                      +----------------+
                      |   (code)       |  Code Node    Companion Node
                      |                |  +-----------+ +-----------+
                      |                |  |           |-|           |
                      +----------------+  |  source   |24|  preview |
                                          |           |px|           |
                                          +-----------+ +-----------+
```

1. First generated node: x = trigger.x + trigger.width + 40px, y = trigger.y (top-aligned)
2. Subsequent nodes: same x, y = previous node bottom + 40px
3. Companion nodes: x = code node.x + code node.width + 24px, y = code node.y (top-aligned)
4. Collision fallback order: Right -> slide down within right column -> Below trigger -> Left -> Above (D-11)

## Companion Node Interaction States

| State | Visual | Duration |
|-------|--------|----------|
| Code streaming | Parent code node has `.canvas-ai-node--streaming` pulse | Until code `</node>` boundary |

## Accessibility Notes

- **Settings tab:** Inherits Obsidian's built-in keyboard navigation and screen reader support via the `Setting` class. No custom ARIA needed.
- **Companion iframe:** The sandboxed iframe is opaque to screen readers. Companion nodes should include `aria-label="Code preview"` on the iframe element for basic accessibility.
- **Reset confirmation:** If using Obsidian `Modal`, keyboard focus is trapped in the modal (Obsidian default behavior).
- **Canvas nodes:** Obsidian canvas nodes do not have robust screen reader support natively. No additional accessibility contract beyond Obsidian's baseline.
